# 制度檔維護協議

寫於 2026-07-10。對象：未來所有 session 的模型（含 Sonnet/Haiku 等級）。
目的：讓制度檔跟著現實走，而不是變成第二份會腐爛的文件（diagnosis.md §1 的教訓）。

## 不用問，直接改（但要遵守格式）

- **事實漂移修正**：檔名、路徑、指令、頁面清單與現實不符 → 直接改，
  前提是你**當下驗證過**新事實（ls / grep / 實跑），不是憑推測。
- **踩坑教訓**：新踩到的坑寫進本檔底部的 LESSONS 區，格式見下。
- **TODO.md 勾選**：完成的事當輪就勾，附一行完成日期。
- **CLAUDE.md「環境實況」段**：驗證過的新事實可直接增修該段落。

## 先問使用者才能動

- 改變任何規則的**語意**（放寬成本閘門、改升降級條件、刪掉一條判準）
- 刪除整個制度檔或從 CLAUDE.md 路由表移除條目
- 把「先問使用者」清單裡的項目移到「直接改」清單（本協議的自我修改要最高門檻）
- 任何讓規則變得**對模型更寬鬆**的修改——這是制度退化的主要路徑，
  預設懷疑自己想放寬規則的動機

## 修改程序（每次都做）

1. 改前備份：`cp {檔案} docs/claude/backups/{檔名}.{YYYY-MM-DD}.bak`
   （同一天多次修改同一檔，備份一次即可）。
2. 改完 read-back 驗證檔案完整。
3. 若改的是 CLAUDE.md 路由表指向的檔名/路徑，同輪更新路由表。

## LESSONS 格式

一條一行事件，寫在本檔最底部，新的加在最上面：

```
- [YYYY-MM-DD] 情境：{一句話} → 教訓：{一句話} → 動作：{改了哪檔哪段 / 或「僅記錄」}
```

判準：值得寫的教訓是「下個 session 沒讀到就會重踩」的坑；
本 session 特有的一次性狀況（某次網路抖動）不值得寫。

## 精簡觸發條件

- LESSONS 超過 20 條，或本檔超過 150 行 → 做一次合併：
  重複主題的教訓收斂成一條規則，升格寫進對應制度檔（diagnosis / dispatch / judgment），
  LESSONS 裡刪掉已升格的條目，並在該條規則後註記來源日期。
- 合併屬於「改規則語意」嗎？——把多條教訓忠實收斂不算，趁機放寬才算。
  拿不準就問使用者。
